# FastAPI Workers 가이드

참고: https://fastapi.tiangolo.com/deployment/server-workers/

---

## 1. 워커란?

uvicorn 프로세스를 여러 개 띄우는 것.
프로세스 1개 = 워커 1개 = 이벤트루프 1개 = CPU 코어 1개 사용.

```
워커 1개                     워커 4개
프로세스 1개                  부모 프로세스 (manager)
└── 이벤트루프 1개            ├── 워커1 (프로세스 + 이벤트루프)
    ├── Task (요청1)          ├── 워커2 (프로세스 + 이벤트루프)
    ├── Task (요청2)          ├── 워커3 (프로세스 + 이벤트루프)
    └── Task (요청3)          └── 워커4 (프로세스 + 이벤트루프)
```

---

## 2. 이벤트루프와 워커의 차이

| | 이벤트루프 | 워커 |
|--|--|--|
| 해결하는 문제 | I/O 대기 (DB, 네트워크) | CPU 코어 활용 |
| 단위 | 프로세스 내부 | 프로세스 자체 |
| 개수 | 워커당 1개 | 서버 코어 수만큼 설정 |
| 관계 | 워커 1개 안에 이벤트루프 1개 | - |

**이벤트루프** = "기다리는 동안 다른 거 해"
**워커** = "일손 자체를 늘려"

---

## 3. 같은 예시로 비교

### 요청 4개 동시에 들어오는 경우

#### I/O 작업 (DB 쿼리, await 있음)

```
워커 1개
이벤트루프
0s  요청1 → await DB → 일시정지
    요청2 → await DB → 일시정지
    요청3 → await DB → 일시정지
    요청4 → await DB → 일시정지
1s  요청 1, 2, 3, 4 동시 완료  → 총 1초

워커 2개
워커1 이벤트루프        워커2 이벤트루프
0s  요청1 → await DB   요청3 → await DB
    요청2 → await DB   요청4 → await DB
1s  완료               완료              → 총 1초
```

**I/O 작업은 워커를 늘려도 차이 없음. 이벤트루프가 이미 처리함.**

---

#### CPU 작업 (계산, await 없음, 요청당 2초)

```
워커 1개 (코어 1개)
0s  요청1 → CPU 점유 (2초)
2s  요청2 → CPU 점유 (2초)   ← 요청1 끝날 때까지 대기
4s  요청3 → CPU 점유 (2초)
6s  요청4 → CPU 점유 (2초)
                              → 총 8초

워커 2개 (코어 2개)
워커1                  워커2
0s  요청1 → CPU 점유   요청3 → CPU 점유
2s  요청2 → CPU 점유   요청4 → CPU 점유
4s  완료               완료              → 총 4초
```

**CPU 작업은 워커를 늘리면 코어를 병렬로 써서 빨라짐.**

## 4. 워커 개수는 동적으로 바뀌지 않음

시작할 때 지정한 수로 고정. CPU 가 바쁘든 한가하든 워커 수는 그대로.

```bash
uvicorn main:app --workers 4  # 항상 4개, 부하에 따라 변하지 않음
```

요청이 워커 수보다 많으면 나머지는 대기함.

```
요청1 → 워커1 (처리 중)
요청2 → 워커2 (처리 중)
요청3 → 워커1, 2 모두 바쁨 → 대기
```

동적으로 늘리려면 Kubernetes HPA 같은 별도 도구가 필요함.

## 5. 단일 서버 vs 클러스터

**단일 서버** = 물리 서버 1대, VM 1개, EC2 1개 등 하나의 머신.

```
단일 서버 (VM 1개)
└── uvicorn --workers 4  ← 코어 4개 활용

클러스터 (VM 여러 개)
├── VM1 → 컨테이너 → uvicorn (워커 1개)
├── VM2 → 컨테이너 → uvicorn (워커 1개)
└── VM3 → 컨테이너 → uvicorn (워커 1개)
         ↑ k8s 가 VM 간 복제/분배 담당
```

---

## 6. Kubernetes 에서는 워커 1개 권장

k8s 에서도 `--workers 4` 는 가능하지만 권장하지 않음.

**이유:** k8s 는 파드(컨테이너) 단위로 스케일링하기 때문.

```
워커 1개짜리 컨테이너 (권장)
├── 파드1 (프로세스 1개)
├── 파드2 (프로세스 1개)   ← k8s 가 파드 수로 제어 (명확)
└── 파드3 (프로세스 1개)

워커 4개짜리 컨테이너 (비권장)
├── 파드1 (프로세스 4개)
├── 파드2 (프로세스 4개)   ← k8s 가 제어하기 복잡
└── 파드3 (프로세스 4개)     파드 죽으면 프로세스 4개 한꺼번에 손실
```

| | 워커 1개 | 워커 4개 |
|--|--|--|
| 스케일링 단위 | 파드 1개 = 프로세스 1개 (명확) | 파드 1개 = 프로세스 4개 (복잡) |
| 리소스 계산 | 쉬움 | 어려움 |
| 파드 장애 시 | 프로세스 1개 손실 | 프로세스 4개 한꺼번에 손실 |
| CPU 활용 | k8s 가 파드 수로 해결 | 컨테이너 안에서 해결 |

---

## 7. 언제 워커를 쓰나?

| 환경 | 워커 설정 | 이유 |
|--|--|--|
| 로컬 개발 | 1개 (기본값) | 불필요 |
| 단일 서버 (VM 1대) | CPU 코어 수만큼 | 멀티코어 활용 |
| Docker Compose | 코어 수만큼 | 컨테이너 1개에서 최대 활용 |
| Kubernetes | 1개 | k8s 가 파드 단위로 스케일링 |

---

## 8. 설정 방법

```bash
# CLI
uvicorn main:app --host 0.0.0.0 --port 8000 --workers 4
```

```python
# main.py 안에서 (uv run -m main 으로 실행 시)
if __name__ == "__main__":
    uvicorn.run("main:app", host="0.0.0.0", port=8000, workers=4)
```

> `workers` 는 `reload=True` 와 함께 사용 불가 (개발 모드와 병행 불가).
