# 이터레이터 → 제너레이터 → 코루틴(async/await) 발전 과정

---

## 1. 이터레이터 - 메모리 문제 해결

**문제:** 리스트는 데이터 전체를 메모리에 올림.

```python
# 100만개 전부 메모리에 올라감
nums = [i for i in range(1_000_000)]
```

**해결:** 이터레이터 - 하나씩 꺼내서 사용.

```python
it = iter([1, 2, 3])
next(it)  # 1
next(it)  # 2
next(it)  # 3
```

**단점:** 직접 만들려면 클래스에 `__iter__`, `__next__` 를 구현해야 해서 번거로움.

---

## 2. 제너레이터 - 이터레이터를 쉽게

**문제:** 이터레이터 직접 구현이 너무 복잡.

```python
# 이터레이터 직접 구현
class MyRange:
    def __iter__(self): ...
    def __next__(self): ...
```

**해결:** `yield` 키워드로 함수 안에서 간단하게 이터레이터 생성.

```python
def my_range(n):
    i = 0
    while i < n:
        yield i   # 값 반환 후 일시정지, next() 호출 시 다음 줄부터 재개
        i += 1

for x in my_range(3):
    print(x)  # 0, 1, 2
```

**yield의 동작:**

```python
def gen():
    print("A")
    yield 1      # 멈추고 1 반환
    print("B")
    yield 2      # 멈추고 2 반환

g = gen()
next(g)  # "A" 출력, 1 반환
next(g)  # "B" 출력, 2 반환
```

함수가 **상태를 기억한 채로 멈출 수 있다**는 게 핵심.

## 3. 동시성 문제 - I/O 대기

파이썬은 GIL(Global Interpreter Lock) 때문에 **싱글스레드**로 동작한다.
한 번에 하나의 작업만 실행할 수 있어서, I/O 대기 중에는 아무것도 못 한다.

```python
import time

def db_query(name):
    print(f"{name} 시작")
    time.sleep(2)       # DB 응답 대기 (이 2초 동안 파이썬은 아무것도 못 함)
    print(f"{name} 완료")

start = time.time()
db_query("쿼리1")   # 2초 대기
db_query("쿼리2")   # 2초 대기
db_query("쿼리3")   # 2초 대기
print(f"총 {time.time() - start:.1f}초")  # 6.0초

# 기다리는 2초 동안 다른 쿼리를 실행하면 총 2초로 끝낼 수 있는데...
```

## 4. 제너레이터로 동시성 해결 시도 (Python 3.4)

`yield` 가 함수를 멈출 수 있다는 점에 착안.
"I/O 대기 중에 yield로 멈추고, 다른 함수를 실행하면 되지 않을까?"

Python 3.4에서 `@asyncio.coroutine` + `yield from` 으로 공식화됐다.

```python
# Python 3.4 방식
import asyncio

@asyncio.coroutine           # "이 함수는 비동기 코루틴이다" 표시
def db_query(name):
    print(f"{name} 시작")
    yield from asyncio.sleep(2)  # I/O 대기 중 제어권 양보
    print(f"{name} 완료")


loop = asyncio.get_event_loop()
loop.run_until_complete(asyncio.gather(
    db_query("쿼리1"),
    db_query("쿼리2"),
    db_query("쿼리3"),
))
```

**문제점 (PEP 492 공식 근거):**

1. **제너레이터와 구분 불가** - 코루틴과 일반 제너레이터가 완전히 같은 문법을 사용.
   코드만 보면 이 함수가 값을 생성하는 제너레이터인지, 비동기 코루틴인지 알 수 없음.

2. **리팩토링 시 조용한 오동작** - 코루틴 여부가 함수 body에 `yield`/`yield from` 이
   있냐 없냐로 결정됨. 리팩토링 중 해당 구문이 추가되거나 사라지면 에러 없이 동작이 바뀜.

   ```python
   @asyncio.coroutine
   def fetch():
       # 만약 yield from 을 실수로 지우면?
       # 데코레이터가 있어도 일반 함수처럼 동작 → 에러 없이 잘못 동작
       result = some_io()
       return result
   ```

3. **`with`, `for` 문에서 사용 불가** - `yield` 가 허용되는 위치에서만 비동기 호출 가능.
   `async with`, `async for` 같은 문법을 만들 수 없었음.

   ```python
   # 불가능했던 것들
   with (yield from db.connect()):  # 어색하고 제한적
       ...
   ```

## 5. 코루틴(async/await) 등장 - Python 3.5

불편함을 해소하기 위해 **코루틴 전용 문법** 도입.
코루틴 = `async def` 로 만든 함수. async/await 와 같은 개념이다.

내부 동작은 `yield from` 과 `await` 가 실질적으로 동일하다. (PEP 492 명시)
문법적 명확성을 위해 새로운 키워드로 분리한 것.

| Python 3.4         | Python 3.5+ |
|--------------------|-------------|
| `@asyncio.coroutine` | `async def` |
| `yield from`       | `await`     |

```python
# Python 3.4 - 불명확
@asyncio.coroutine
def db_query(name):
    yield from asyncio.sleep(2)   # 이게 비동기 대기인지 값 생성인지 불명확

# Python 3.5 - 명확
async def db_query(name):         # async def: "이 함수는 코루틴" 명확하게 선언
    await asyncio.sleep(2)        # await: "여기서 I/O 대기" 명확하게 표현
```

```python
import asyncio, time

async def db_query(name):
    print(f"{name} 시작")
    await asyncio.sleep(2)   # 여기서 멈춤 → 이벤트루프가 다른 코루틴 실행
    print(f"{name} 완료")    # I/O 완료 후 재개
    return f"{name} 결과"


# gather: 여러 코루틴을 동시에 실행
# 가장 느린 코루틴이 완료될 때까지 기다린 뒤 다음 줄로 넘어감
# 모든 코루틴이 끝나야 r1, r2, r3 에 결과가 채워지고 print 가 실행됨
async def main():
    start = time.time()
    r1, r2, r3 = await asyncio.gather(
        db_query("쿼리1"),   # 2초
        db_query("쿼리2"),   # 2초  → 셋이 동시에 실행되므로 가장 느린 2초를 기다림
        db_query("쿼리3"),   # 2초
    )
    print(f"총 {time.time() - start:.1f}초")  # 2.0초 (순차였으면 6초)


asyncio.run(main())
```

## 전체 흐름

```
문제                          해결               핵심 키워드
─────────────────────────────────────────────────────────────
메모리에 전부 올라감         → 이터레이터        iter, next
이터레이터 구현 번거로움     → 제너레이터        yield
I/O 대기 중 CPU 낭비         → 제너레이터로 시도  yield from
제너레이터 기반 비동기 불편  → 코루틴(async/await) async, await
```

---

## yield vs await

```
yield   →  제너레이터  →  값을 하나씩 꺼내기 위해 멈춤
await   →  코루틴      →  I/O 기다리는 동안 다른 코루틴에 제어권 양보
```

개념은 동일 - **멈추고 재개**. 목적이 다를 뿐.
