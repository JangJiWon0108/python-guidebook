# 코루틴 (Coroutine) & async/await

## 왜 필요한가?

```python
import time

def fetch_data(name):
    print(f"{name} 요청 시작")
    time.sleep(2)              # DB 조회, 네트워크 요청 등 I/O 대기 (2초)
    print(f"{name} 완료")
    return f"{name} 결과"


start = time.time()

fetch_data("쿼리1")  # 2초 대기
fetch_data("쿼리2")  # 2초 대기
fetch_data("쿼리3")  # 2초 대기

print(f"총 소요시간: {time.time() - start:.1f}초")  # 6.0초

# 문제: 쿼리1이 기다리는 2초 동안 쿼리2, 쿼리3은 아무것도 못 함
# 기다리는 시간을 활용하면 총 2초로 끝낼 수 있는데...
```

---

## 코루틴이란?

`async def` 로 정의한 함수. 제너레이터처럼 **중간에 멈추고 재개할 수 있는 함수**.
코루틴 = async/await. 별개 개념이 아니다.

```python
import asyncio

async def fetch_data(name):
    print(f"{name} 요청 시작")
    await asyncio.sleep(2)     # I/O 대기 - "여기서 기다리는 동안 다른 거 해도 됨"
    print(f"{name} 완료")
    return f"{name} 결과"


# 코루틴 함수를 호출해도 즉시 실행되지 않음. 코루틴 객체만 반환됨.
coro = fetch_data("쿼리1")
print(type(coro))   # <class 'coroutine'>

# 실행하려면 이벤트루프가 필요함
asyncio.run(fetch_data("쿼리1"))
```

---

## await 키워드

`await` 를 만나면 현재 코루틴이 일시정지되고, 이벤트루프가 다른 코루틴을 실행한다.
I/O가 완료되면 이벤트루프가 다시 이 코루틴으로 돌아온다.

```python
async def main():
    print("시작")
    result = await fetch_data("쿼리1")  # fetch_data 완료될 때까지 기다림
                                         # 기다리는 동안 이벤트루프는 다른 코루틴 실행 가능
    print(result)
    print("끝")
```

`await` 뒤에는 **어웨이터블(Awaitable)** 만 올 수 있다.
- 코루틴 (`async def` 함수 호출 결과)
- `asyncio.Task`
- `asyncio.Future`

## 순차 실행 vs 동시 실행

```python
import asyncio
import time

async def fetch_data(name, delay=2):
    print(f"{name} 시작")
    await asyncio.sleep(delay)  # I/O 대기 시뮬레이션
    print(f"{name} 완료")
    return f"{name} 결과"


# 순차 실행 - 총 6초
async def sequential():
    start = time.time()

    r1 = await fetch_data("쿼리1")  # 2초 기다린 후 다음으로
    r2 = await fetch_data("쿼리2")  # 2초 기다린 후 다음으로
    r3 = await fetch_data("쿼리3")  # 2초 기다린 후 끝

    print(f"순차 소요시간: {time.time() - start:.1f}초")  # 6.0초


# 동시 실행 - 총 2초
async def concurrent():
    start = time.time()

    # gather: 여러 코루틴을 동시에 실행
    # 가장 느린 코루틴이 완료될 때를 기준으로 기다림
    # 모든 코루틴이 끝나야 r1, r2, r3 에 결과가 채워지고 다음 줄로 넘어감
    r1, r2, r3 = await asyncio.gather(
        fetch_data("쿼리1"),
        fetch_data("쿼리2"),
        fetch_data("쿼리3"),
    )

    print(f"동시 소요시간: {time.time() - start:.1f}초")  # 2.0초


asyncio.run(sequential())
asyncio.run(concurrent())
```

## 이벤트루프

코루틴들을 번갈아가며 실행하는 스케줄러.

```
이벤트루프 동작 방식:

1. fetch("쿼리1") 시작
2. await asyncio.sleep(2) 만남 → 쿼리1 일시정지
3. fetch("쿼리2") 시작
4. await asyncio.sleep(2) 만남 → 쿼리2 일시정지
5. fetch("쿼리3") 시작
6. await asyncio.sleep(2) 만남 → 쿼리3 일시정지
7. 2초 후 쿼리1 완료 → 재개
8. 쿼리2 완료 → 재개
9. 쿼리3 완료 → 재개
```

```python
# asyncio.run() 이 이벤트루프를 생성하고 코루틴을 실행함
asyncio.run(main())

# 이벤트루프는 스레드 하나에서 동작 (싱글스레드)
# 멀티스레드가 아니라 "기다리는 시간을 활용"하는 방식
```

## gather vs create_task

### gather - 한꺼번에 실행하고 전부 기다림

```python
import asyncio, time

async def fetch(name, delay):
    print(f"{name} 시작")
    await asyncio.sleep(delay)
    print(f"{name} 완료")
    return f"{name} 결과"


async def main():
    start = time.time()

    # gather 에 넘기는 순간 동시 실행 시작
    # 가장 느린 코루틴 기준으로 전부 끝날 때까지 기다림
    # 결과는 넘긴 순서대로 r1, r2, r3 에 담김
    r1, r2, r3 = await asyncio.gather(
        fetch("쿼리1", delay=2),
        fetch("쿼리2", delay=1),
        fetch("쿼리3", delay=3),  # 가장 느림 → 3초 기준으로 기다림
    )

    print(r1, r2, r3)
    print(f"총 {time.time() - start:.1f}초")  # 3.0초 (순차였으면 6초)


asyncio.run(main())
```

---

### create_task - 등록 즉시 실행, 결과는 나중에

```python
async def main():
    start = time.time()

    # create_task: 이벤트루프에 등록 즉시 백그라운드에서 실행 시작
    task1 = asyncio.create_task(fetch("쿼리1", delay=2))
    task2 = asyncio.create_task(fetch("쿼리2", delay=1))

    # task 등록 후 다른 작업 가능 (이 사이에도 task는 실행 중)
    print("task 등록 완료, 다른 작업 실행 중...")
    await asyncio.sleep(0.5)  # 다른 작업 시뮬레이션
    print("다른 작업 끝")

    # 나중에 필요할 때 각각 결과를 기다림
    r1 = await task1  # task1 완료될 때까지 기다림
    r2 = await task2  # task2 완료될 때까지 기다림

    print(r1, r2)
    print(f"총 {time.time() - start:.1f}초")


asyncio.run(main())
```

### 비교

| | `gather` | `create_task` |
|--|--|--|
| 실행 시작 | gather 호출 시 | create_task 호출 시 |
| 결과 수집 | 전부 끝나면 한꺼번에 | 각각 await 로 따로 |
| 중간에 다른 작업 | 불가 (await 에서 멈춤) | 가능 |
| 사용 상황 | 결과가 전부 필요할 때 | task 등록 후 다른 로직도 필요할 때 |

대부분은 `gather` 로 충분하고, task 등록 후 중간에 다른 로직이 필요할 때 `create_task` 사용.

```
gather       →  실행 + 대기를 한 번에. await 하는 순간 거기서 멈춰서 기다림.
create_task  →  실행은 먼저, 대기는 내가 원할 때. 중간에 다른 작업 가능.
```

FastAPI 같은 웹 프레임워크는 요청마다 코루틴을 자동으로 처리해주기 때문에
애플리케이션 코드에서 `gather`/`create_task` 를 직접 쓸 일이 많지 않다.
하나의 API 핸들러 안에서 여러 DB 쿼리를 동시에 날려야 할 때 주로 사용한다.

## yield vs await 비교

```python
# 제너레이터 - yield로 멈춤
def gen():
    yield 1   # 값을 반환하고 멈춤. 목적: 값을 하나씩 꺼내기

# 코루틴 - await로 멈춤
async def coro():
    await asyncio.sleep(1)  # I/O 기다리는 동안 멈춤. 목적: 대기 시간 활용
```

개념은 동일 - **멈추고 재개**. 목적이 다를 뿐.

---

## 정리

```
async def   →  코루틴 함수 정의
await       →  I/O 대기 중 이벤트루프에 제어권 넘김
asyncio.run →  이벤트루프 생성 + 코루틴 실행
gather      →  여러 코루틴 동시 실행
create_task →  코루틴을 백그라운드 Task로 등록
```

- **싱글스레드**로 동작. 멀티스레드가 아님.
- I/O 대기 시간이 많은 작업(DB, 네트워크)에 효과적.
- CPU를 많이 쓰는 작업(계산, 이미지 처리)엔 효과 없음 → 멀티프로세싱 사용.

네, 아주 좋은 아이디어입니다!

4가지 케이스 각각의 상황에서 `asyncio.current_task()`(현재 실행 중인 놈)와 `asyncio.all_tasks()`(장부에 등록된 모든 놈)를 찍어보면, 이벤트 루프가 내부적으로 어떻게 장부를 관리하고 제어권을 넘기는지 눈으로 완벽하게 확인할 수 있습니다.

이해를 돕기 위해 **각 케이스별로 상태를 실시간으로 스냅샷 찍듯 출력하는 완성형 코드**를 준비했습니다.

### 💻 4가지 케이스 모니터링 전체 코드

일반 함수(케이스 1)에서는 이벤트 루프가 없으므로 `asyncio` 메서드를 쓰면 에러가 납니다. 따라서 비동기 함수(케이스 2, 3, 4)에서만 상태를 정밀 출력하도록 구성했습니다.

```python
import asyncio
import time

# 📊 현재 이벤트 루프의 '스냅샷'을 예쁘게 찍어주는 모니터링 함수
def print_loop_status(checkpoint_name):
    try:
        current = asyncio.current_task()
        current_name = current.get_name() if current else "None"
    except RuntimeError:
        # 이벤트 루프가 없는 일반 동기 환경일 때
        print(f"\n[🚨 {checkpoint_name}] 현재 이벤트 루프가 실행 중이 아닙니다 (일반 함수 환경).")
        return

    all_tasks = asyncio.all_tasks()
    task_names = [t.get_name() for t in all_tasks]
    
    print(f"\n📌 [{checkpoint_name}]")
    print(f" └ 🏃‍♂️ 현재 제어권을 쥔 태스크: {current_name}")
    print(f" └ 📊 현재 장부(Loop)에 있는 총 태스크 [{len(all_tasks)}개]: {task_names}")


# 🛠️ 테스트용 비동기 작업 함수
async def fetch(name):
    print_loop_status(f"{name} 내부 - sleep 직전")
    await asyncio.sleep(1)  # 여기서 제어권 양보 발생!
    print_loop_status(f"{name} 내부 - sleep 직후")
    return f"{name} 결과"


# ==========================================
# 케이스 1: 일반 함수 순차 실행
# ==========================================
def run_case1():
    print("\n" + "="*50 + "\n🎬 케이스 1: 일반 함수 순차 실행 시작\n" + "="*50)
    print_loop_status("케이스 1 시작점")
    time.sleep(1)
    time.sleep(1)
    print("✅ 케이스 1 완료 (총 2초)")


# ==========================================
# 케이스 2: async 순차 실행 (await 각각)
# ==========================================
async def run_case2():
    print("\n" + "="*50 + "\n🎬 케이스 2: async 순차 실행 시작\n" + "="*50)
    print_loop_status("케이스 2 시작점")
    
    # 세밀하게 모니터링하기 위해 이름을 지정해서 실행
    await fetch("순차_쿼리1")
    await fetch("순차_쿼리2")
    print("✅ 케이스 2 완료 (총 2초)")


# ==========================================
# 케이스 3: create_task 개별 실행 (동시)
# ==========================================
async def run_case3():
    print("\n" + "="*50 + "\n🎬 케이스 3: create_task 개별 동시 실행 시작\n" + "="*50)
    print_loop_status("케이스 3 시작점 (태스크 등록 전)")
    
    # 백그라운드에 등록 (등록만 하고 바로 아랫줄로 넘어감!)
    t1 = asyncio.create_task(fetch("태스크_쿼리1"), name="Task-1")
    t2 = asyncio.create_task(fetch("태스크_쿼리2"), name="Task-2")
    
    print_loop_status("케이스 3 (태스크 2개 등록 직후, await 전)")
    
    # 이제 결과를 기다리며 제어권을 넘김
    await t1
    await t2
    print("✅ 케이스 3 완료 (총 1초)")


# ==========================================
# 케이스 4: asyncio.gather 그룹 실행 (동시)
# ==========================================
async def run_case4():
    print("\n" + "="*50 + "\n🎬 케이스 4: asyncio.gather 그룹 동시 실행 시작\n" + "="*50)
    print_loop_status("케이스 4 시작점 (gather 실행 전)")
    
    # gather는 내부적으로 자동 명칭을 부여하므로, 구조를 보기 위해 바로 호출
    await asyncio.gather(
        fetch("가더_쿼리1"),
        fetch("가더_쿼리2")
    )
    print("✅ 케이스 4 완료 (총 1초)")


# 🚀 메인 실행부
async def main():
    # 1번은 일반 함수이므로 루프 밖에서 따로 실행하기 위해 main 위에서 제어
    pass

if __name__ == "__main__":
    # 케이스 1 실행 (루프 없음)
    run_case1()
    
    # 케이스 2, 3, 4 실행 (루프 안에서 실행)
    async def loop_main():
        # 가독성을 위해 각 태스크(함수) 이름을 메인으로 지정
        asyncio.current_task().set_name("Main-Loop")
        await run_case2()
        await run_case3()
        await run_case4()
        
    asyncio.run(loop_main())

```

### 🧐 핵심 출력 결과 분석 (이것만 이해하면 끝납니다!)

코드를 돌렸을 때 나오는 로그 중 **가장 중요한 결정적 차이점**만 짚어드릴게요.

#### 📍 케이스 2 (async 순차)의 장부 상태

```text
📌 [순차_쿼리1 내부 - sleep 직전]
 └ 🏃‍♂️ 현재 제어권을 쥔 태스크: Main-Loop
 └ 📊 현재 장부(Loop)에 있는 총 태스크 [1개]: ['Main-Loop']

```

> **분석**: `await fetch()`로 호출하면 새로운 독립 태스크가 생기지 않고, `Main-Loop`가 직접 들어가서 실행합니다. 그래서 장부에는 늘 `Main-Loop` **단 1개**만 존재하고, 쿼리1이 완전히 끝나 지워져야 비로소 쿼리2가 장부에 올라옵니다.

#### 📍 케이스 3 (create_task)의 장부 상태 🌟 (가장 중요)

```text
📌 [케이스 3 (태스크 2개 등록 직후, await 전)]
 └ 🏃‍♂️ 현재 제어권을 쥔 태스크: Main-Loop
 └ 📊 현재 장부(Loop)에 있는 총 태스크 [3개]: ['Main-Loop', 'Task-1', 'Task-2']

```

> **분석**: 보이시나요? `await`로 기다리기도 전인데, `create_task`를 하자마자 장부에 `Task-1`과 `Task-2`가 **동시에 대기 등록(총 3개)** 되었습니다!
> 이 상태에서 `Main-Loop`가 `await t1`을 외치며 자리를 비워주면, 이벤트 루프가 장부에 있던 `Task-1`과 `Task-2`를 빛의 속도로 번갈아 실행시키기 때문에 시간이 절반으로 줄어드는 것입니다.

#### 📍 케이스 4 (asyncio.gather)의 장부 상태

```text
📌 [가더_쿼리1 내부 - sleep 직전]
 └ 🏃‍♂️ 현재 제어권을 쥔 태스크: Main-Loop
 └ 📊 현재 장부(Loop)에 있는 총 태스크 [4개]: ['Main-Loop', 'Task-3', 'Task-4', 'Task-5']

```

> **분석**: `gather`를 쓰면 내부적으로 관리용 태스크(`Task-3`)와 우리가 넣은 함수 2개(`Task-4`, `Task-5`)를 **알아서 장부에 한 번에 등록**하고 동시에 굴리는 것을 볼 수 있습니다. (결국 3번과 원리가 똑같다는 증거입니다.)
